## Architecture Decisions

- globals.css defines shadcn's standard tokens (background/foreground/primary/etc.) plus custom semantic tokens (bg-base, bg-surface, bg-elevated, bg-subtle, border-surface-border, border-surface-border-subtle, text-copy-primary/secondary/muted/faint, text-brand, bg-accent-dim, bg-ai/text-ai-text, bg-error/success/warning) per context/ui-context.md, both pointing at the same dark hex palette. `<html>` carries a permanent `dark` class since the app has no light mode.
- This repo's Next.js (16.2.11) and Clerk (`@clerk/nextjs` 7.5.22, `@clerk/ui` 1.25.6) are both ahead of general training data — confirmed against `node_modules/next/dist/docs` and the installed packages' own `.d.ts` files rather than assumed. Notably: `middleware.ts` is renamed to `proxy.ts` (same runtime behavior, `next.config` matcher unchanged); Clerk's appearance API uses `theme:` (not the older `baseTheme:`) inside `appearance={{ theme, variables }}`.
- Route protection uses `createRouteMatcher` + `auth.protect()` in `proxy.ts`, per the auth feature spec. Clerk's own SDK logs a deprecation warning for `createRouteMatcher` in favor of resource-based (per-page/action) auth checks — still functional, not removed, but worth revisiting if this Clerk version bumps further.
- Fixed a pre-existing bug in globals.css: `--font-sans` was defined as `var(--font-sans)` (circular, self-referencing), so the `font-sans` Tailwind utility never actually resolved to Geist Sans anywhere in the app — it silently fell back to the browser default font. Now maps to `var(--font-geist-sans)` (and added `--font-mono: var(--font-geist-mono)` alongside it). This affects all pages, not just auth. Clerk's `appearance.variables.fontFamily`/`fontFamilyMono` were also pinned to the same Geist variables since Clerk's portal-rendered UI doesn't reliably inherit body font otherwise.
- Fixed a bug in `prisma.config.ts`: it only did `import "dotenv/config"`, which loads `.env` — but `DATABASE_URL` (and all other secrets) live in `.env.local` (Next.js's convention, confirmed by `next build`'s own "Environments: .env.local, .env" line). The Prisma CLI has no built-in awareness of `.env.local`, so `prisma migrate dev` failed with "datasource.url property is required" even though the app itself worked fine. Fixed by explicitly loading `.env.local` with `override: true` after the default `dotenv/config` import.
- Prisma 7.9.0's generated client (`prisma-client` provider, output already pointed at `app/generated/prisma` before this feature) accepts either `adapter` or `accelerateUrl` in its constructor — mutually exclusive at the type level. Accelerate no longer needs the separate `@prisma/extension-accelerate` package for basic connectivity in this version; only add that package later if caching (`cacheStrategy`) is needed.
- `proxy.ts`'s `isPublicRoute` matcher now includes `/api(.*)`, exempting all API routes from middleware-level `auth.protect()`. Discovered while building 06-project-apis: `auth.protect()` in `clerkMiddleware` 307-redirects unauthenticated requests to `/sign-in` (confirmed via curl, header `x-clerk-auth-reason: dev-browser-missing`) instead of returning `401` JSON, which the spec required. Clerk's own SDK already prints a deprecation warning for `createRouteMatcher` favoring resource-based (per-route) auth checks over middleware path-matching — this change follows that: every API route handler now calls `auth()` itself and returns `401`/`403` explicitly. Any future API route must do its own `auth()` check; it is no longer protected by the proxy.
- Project ids are now client-generated (`slugify(name)-{shortSuffix}`) and sent to `POST /api/projects`, which uses them as the literal `Project.id` (falling back to Prisma's `cuid()` default only if the client omits/mis-formats it). This is a deliberate divergence from 06-project-apis's "use the schema's existing ID strategy" note, which was guarding against sequential/auto-increment ids, not client-supplied ones. Reasoning: `architecture-context.md` states canvas snapshots live at `canvas/{projectId}.json` and that "Liveblocks room tokens are issued only after verifying project membership" — there's no separate `roomId` field anywhere in the schema, so the project id *is* the Liveblocks room id by construction. A human-readable, collision-resistant id (slug + suffix) makes that shared id useful as a room/URL identifier instead of an opaque cuid. If a future spec explicitly wants opaque server-generated ids instead, this is the code to revisit: `hooks/use-project-actions.ts`'s `roomId` computation and the `id` handling in `app/api/projects/route.ts`'s `POST`.
- Server-side project fetching for the sidebar lives in `app/editor/layout.tsx`, not `app/editor/page.tsx`, even though 07-wire-editor-home's spec says "the editor home page is a server component... fetch... pass both lists to the sidebar." The sidebar is rendered by the layout (it wraps every route under `/editor/*`, including the future `/editor/[projectId]` workspace route), not by the home page itself, so the layout is the component that actually needs the data. `app/editor/page.tsx` was left untouched.
- Shared projects are matched via `ProjectCollaborator.email` against the signed-in user's Clerk `primaryEmailAddress`, not a Clerk user id — `ProjectCollaborator` only stores `email` in the schema (collaborators are invited by email before they necessarily have an account). If a user's primary email changes, or a collaborator is invited under a non-primary email, they won't see the shared project — acceptable for now, worth revisiting if collaboration becomes a bigger feature.
